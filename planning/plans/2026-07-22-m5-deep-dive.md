**Content map (from spec — the deep-dive intent):**

1. **Chunk size & overlap, felt not formal:** analogy — cutting a long document into index cards; too
   small and each card loses context (a sentence orphaned from its paragraph), too large and the card
   holds too many unrelated ideas for the embedding to represent well; overlap is photocopying the last
   few lines of each card onto the next so a fact split across a page boundary is never orphaned. Mermaid:
   document → chunks (with overlap band) → embeddings → vector store.
2. **top-k:** how many index cards you hand the model per question; too few misses the answer, too many
   dilutes the context window and can confuse the model with near-duplicates or off-topic chunks.
3. **Similarity metric & embedding model choice:** why the embedding model (`nomic-embed-text` in this
   course) determines what "similar" even means; cosine similarity intuition; why swapping embedding
   models means re-embedding the whole corpus (embeddings from different models aren't comparable).
4. **Context-window budget arithmetic:** tokens spent on system prompt + retrieved chunks + question +
   headroom for the answer; a concrete worked budget using this course's `qwen2.5:1.5b` context length
   and this lab's actual chunk size, so learners can size top-k × chunk-size against a real ceiling.
5. **Failure modes:** retrieval returning technically-similar-but-wrong-answer chunks (embedding
   similarity ≠ relevance); why naive RAG (this course's M5) has no re-ranking step and what that costs.
6. **Evaluating retrieval quality:** eyeballing retrieved chunks against a known-answer question set —
   simple, no eval framework required for this course's scope.
7. **Experiment (the lab part):** re-ingest the M5 corpus with different chunking parameters and compare:
   - Baseline: the lab's own ingestion (whatever chunk size/overlap `labs/m5` currently uses — confirm
     exact values from `labs/m5/app` source during Task 2 probe).
   - Variant A: smaller chunks, no overlap.
   - Variant B: larger chunks, generous overlap.
   - Use separate ChromaDB collection names per variant (same running container, same embedding model)
     to avoid doubling memory/process count.
   - Ask a FIXED set of 3-4 questions from the existing corpus (reuse or lightly extend the questions
     already used in `labs/m5/rag_roundtrip.py` if present) against each variant's collection.
   - Compare table: variant, retrieved chunk snippets (short excerpts), whether the retrieved context
     actually contained the answer, and the model's final answer for each.
8. **Where you will use this:** sizing chunk/overlap/top-k for a real corpus (short FAQ vs long runbook);
   recognizing a "wrong chunk retrieved" failure vs a "model ignored good context" failure; when naive
   RAG's lack of re-ranking will start to hurt.
9. **Teardown:** drop the two variant ChromaDB collections (keep the baseline collection the core lab
   depends on); note this does NOT touch the running `labs/m5` app.
