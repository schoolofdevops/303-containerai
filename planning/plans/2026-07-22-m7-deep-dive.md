**Content map (from spec — the deep-dive intent):**

1. **Temperature, felt not formal:** analogy — a musician improvising: low temperature plays it safe and
   close to the melody every time (deterministic, repeatable), high temperature takes more liberties
   (creative but riskier, sometimes off-key). Connect to why an Incident Crew agent triaging a real outage
   wants LOW temperature (repeatable, boring, correct) while a brainstorming agent might want higher.
2. **Max iterations / tool-loop control:** why an agent given a tool and no iteration cap can loop
   forever (or until timeout) re-calling a tool with slightly different arguments, hoping for a better
   result. Analogy: someone re-googling the same question with tiny rewordings instead of stopping to
   think. Mermaid: agent → tool call → observe result → decide (answer vs retry) → cap enforced.
3. **Delegation between agents:** how the Incident Crew's agents hand off work (who decides an
   Investigator's finding is "done" and ready to pass to a Responder); the cost of delegation depth (each
   hop is another LLM call, more latency, more chances to drift off-task).
4. **Guardrail placement:** where in the pipeline a guardrail check should sit — before a tool call
   (validate arguments), after a tool call (validate results before trusting them), or at final-answer
   time (validate output before it reaches a human) — reference M8's guardrail eval as the sibling
   concept (governance) vs this module's knob-tuning (behavior).
5. **Small-model variance (explicit caveat):** `qwen2.5:1.5b`-class models respond to temperature/prompt
   changes less predictably than larger models — results in this deep dive are illustrative of the
   MECHANISM, not a universal recipe; note what changes (or doesn't) if the learner swaps in a larger
   model later.
6. **Experiment (the lab part):** run the Incident Crew with knob variants and observe:
   - Baseline: the lab's own crew run (`docker compose run --rm crew "..."` with default config/knobs —
     confirm exact defaults from `labs/m7/crew` source during Task 2 probe).
   - Variant A: low temperature (near-deterministic) — same incident prompt.
   - Variant B: high temperature — same incident prompt.
   - Optionally Variant C: reduced max-iterations if the crew config exposes it, to observe truncated
     reasoning/tool loops.
   - Run sequentially (never parallel — CPU/memory budget + shared ChromaDB container).
   - Compare table: variant, wall time, number of tool calls observed, final recommendation text
     (verbatim excerpt), qualitative note on consistency vs the baseline.
7. **Where you will use this:** setting temperature per-agent-role in a real crew (triage vs
   brainstorming); capping iterations to bound cost/latency in production; deciding where a guardrail
   check earns its keep.
8. **Teardown:** no persistent state beyond the shared `labs/m7` ChromaDB container (already covered by
   `labs/m7/down.sh`); note the experiment's variant transcripts are local files only, safe to delete.
