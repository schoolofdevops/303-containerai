**Content map (from spec — the deep-dive intent):**

1. **PagedAttention, felt not formal:** the KV cache as paged virtual memory — analogy: a hotel that
   assigns guests (tokens) to rooms (fixed-size pages) on demand instead of reserving one giant suite per
   guest up front; no wasted rooms, no giant contiguous block required. Contrast with naive contiguous KV
   cache allocation (why it fragments and wastes memory on variable-length sequences). Mermaid: request →
   token generation → KV cache pages allocated/freed per block.
2. **Continuous batching vs static batching:** why waiting for a full batch (static) starves short
   requests behind long ones; how vLLM admits/evicts requests into a running batch mid-flight
   (iteration-level scheduling). Analogy: a diner seating parties as tables free up vs waiting for the
   whole restaurant to empty before reseating anyone.
3. **What our lab flags actually did** — walk `labs/m3/compose.yaml` / `Dockerfile` line by line: why
   `dtype float32` (bf16 unsupported on this CPU path — arm64 Rancher VM), what `swap-space` buys you
   (CPU-RAM overflow for KV cache pages when GPU/accelerator memory would normally hold them — on
   CPU-only this maps to process RAM), `max-model-len` (context cap vs memory ceiling trade-off), and the
   CPU core cap set in `up.sh`'s VM-size guard (4 CPU / 6 GB minimum). Reference the 5 arm64/Rancher
   fixes already documented in `planning/lab-tests/m3.md`.
4. **Throughput levers:** batch size ceiling, `max-num-seqs`, why CPU inference is latency-bound
   differently than GPU (no tensor-core parallelism ceiling to hit first — memory bandwidth dominates).
5. **When PagedAttention-class serving matters vs when it's overkill:** single-user local dev
   (Ollama-class servers are fine) vs multi-tenant/concurrent-request production serving (vLLM-class
   matters) — decision Mermaid.
6. **Experiment (the lab part):** small throughput/latency comparison, vLLM-CPU (existing lab container)
   vs native Ollama, same prompt set, same model family/size as closely as available on both:
   - Fixed prompt set (3-5 prompts, mixed short/long) sent to the running `labs/m3` vLLM-CPU
     endpoint (`http://localhost:8009/v1`, already up per `up.sh`).
   - Same prompts sent to native Ollama (`http://localhost:11434`) with a comparably-sized model.
   - Measure: tokens/sec (or wall time for a fixed max_tokens), P50 latency for the fixed prompt set.
     Single-request comparison only — this lab is not attempting concurrent-load benchmarking (out of
     scope, out of CPU budget on 4 CPU / 6 GB).
   - Compare table: engine, tokens/sec, latency, notes (batching behavior observed, if any, with 2-3
     prompts fired back to back).
7. **Where you will use this:** reading vLLM server flags/docs with intent instead of copy-paste;
   knowing when a project needs vLLM-class serving vs when Ollama is enough; sizing `swap-space` and
   `max-model-len` for a real deployment's memory budget.
8. **Teardown:** `docker compose down` for the m3 vLLM container (reuses existing `labs/m3/down.sh` —
   note it, don't duplicate); no separate resources created by the experiment beyond a results log file.
